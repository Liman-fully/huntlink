## 解决方案详解

### 1. Express 类型导入
```typescript
// 修复前
import { Injectable } from '@nestjs/common';

async uploadResume(file: Express.Multer.File) // ❌ 错误：找不到命名空间

// 修复后
import * as Express from 'express';
import { Injectable } from '@nestjs/common';

async uploadResume(file: Express.Multer.File) // ✅ 正确
```

### 2. JWT_SECRET 安全修复
```typescript
// 修复前
useFactory: (configService: ConfigService) => ({
  secret: configService.get<string>('JWT_SECRET', 'huntlink-jwt-secret-key-2026'),
  //                         ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
  //                         硬编码默认值，不安全
})

// 修复后
useFactory: (configService: ConfigService) => {
  const secret = configService.get<string>('JWT_SECRET');
  if (!secret) {
    throw new Error('JWT_SECRET environment variable is required');
  }
  return { secret, signOptions: { expiresIn: '7d' } };
}
```

## 经验总结

### 1. TypeScript 类型导入最佳实践
- 使用第三方库的类型时，确保安装对应的 `@types/*` 包
- `Express.Multer.File` 需要同时安装 `@types/multer` 和导入 `express` 命名空间
- 全局类型（如 `describe`, `it`）需要在 tsconfig 或 ESLint 中配置

### 2. NestJS 配置安全性
- 敏感配置（JWT_SECRET、数据库密码等）必须从环境变量读取
- 不要提供默认值，避免开发环境使用弱密钥
- 启动时验证必需的环境变量，失败时快速抛出错误

### 3. 前端依赖管理
- Node.js 版本过高可能导致依赖安装问题
- 生产项目建议使用 Node.js LTS 版本（当前是 v20.x）
- 遇到依赖问题时，尝试 `npm cache clean --force` 后重新安装

### 4. 代码审查流程
- 先编译检查（`tsc --noEmit`），发现类型错误
- 再代码质量检查（ESLint），发现代码坏味道
- 最后功能测试，确保修复不破坏现有功能
