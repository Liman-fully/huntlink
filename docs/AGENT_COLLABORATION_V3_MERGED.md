## 零、组织架构（统一版）

### 0.1 猎脉三部制

**总领**：司命大人（总揽全局，战略决策）

**三大部门**：

**🏛️ 天策府（左护法）**
- **主官**: 左护法
- **职责**: 产品策略、方案审核、部署验收、协作体系维护
- **子 Agent**: 策画郎中（产品）、审议郎中（审核）、验收郎中（部署）
- **汇报**: 直接向司命大人汇报

**⚔️ 神机营（Claw/都统）**
- **主官**: 都统（Claw）
- **职责**: 技术实现、代码开发、问题修复、部署运维
- **子 Agent**: 名字、职称和工作由都统根据任务自定义
- **汇报**: 直接向司命大人汇报

**🛡️ 镇抚司（右护法）**
- **主官**: 右护法
- **职责**: 技术实现、代码开发、问题修复、部署运维（和神机营一样）
- **子 Agent**: 名字、职称和工作由右护法根据任务自定义
- **汇报**: 直接向司命大人汇报

### 0.2 组织层级

```
一级：司命大人（总领）
         ↓
二级：三大主官（左护法、都统、右护法）→ 直接向司命大人汇报
         ↓
三级：子 Agent → 向各自主官汇报，名字、职称和工作由各主官自定义
```

### 0.3 协作机制

| 部门 | 职责 | 工作性质 | 协作方式 |
|------|------|---------|---------|
| **天策府** | 决定"做什么" | 管理 + 协调 | 任务分配、方案审核、部署验收 |
| **神机营** | 决定"怎么做" | 技术开发 | 并行开发团队 1 |
| **镇抚司** | 决定"怎么做" | 技术开发 | 并行开发团队 2 |

**说明**: 神机营和镇抚司都是开发团队，并行工作，互相配合，共同向司命大人汇报。

### 0.4 信息共享机制

- **共享文档**: MEMORY.md、每日日志、协作规范（三方可写）
- **横向协作**: 通过文档标注 `@部门` 进行跨部门沟通
- **状态同步**: 每日更新 `.ai-collab-status.md`

## 一、核心协作流程（统一版）

### 1.1 技能强制前置检查流程

**适用范围**: 天策府、神机营、镇抚司全体 Agent

```
收到任务
    ↓
【强制】搜索技能（find-skills / clawhub）- 5 分钟
    ↓
【强制】对比选择（安装量、评价、功能）- 5 分钟
    ↓
【强制】安装最优技能 - 10 分钟
    ↓
【强制】更新技能知识库 - 5 分钟
    ↓
使用技能解决问题 - 30 分钟
    ↓
完成 + 记录教训（更新工作记忆）
```

**执行规则**:
- ⚠️ **绝对禁止**: 不搜索技能就自己瞎想
- ⚠️ **时间上限**: 同一问题不超过 30 分钟
- ✅ **强制执行**: 每次任务前自动检查技能库
- ✅ **版本检查**: 每次使用前检查技能更新

### 1.2 子代理任务标准化模板

**适用范围**: 神机营任务分配

```typescript
interface SubAgentTask {
  // 基础信息
  id: string;                    // 任务 ID（自动生成）
  title: string;                 // 任务标题（简洁明确）
  description: string;           // 任务描述（详细）
  
  // 输入输出
  input: {
    files: string[];             // 输入文件列表
    context: string;             // 上下文信息
    constraints: string[];       // 约束条件
  };
  
  output: {
    deliverables: string[];      // 交付物列表
    format: string;              // 交付格式（代码/文档/报告）
    deadline?: string;           // 截止时间（可选）
  };
  
  // 执行配置
  execution: {
    timeout: number;             // 超时时间（分钟）
    parallel: boolean;           // 是否可并行
    dependencies?: string[];     // 依赖任务 ID
  };
  
  // 质量要求（镇抚司审查点）
  quality: {
    acceptanceCriteria: string[]; // 验收标准
    testRequired: boolean;        // 是否需要测试
    documentationRequired: boolean; // 是否需要文档
    codeReviewRequired: boolean;    // 是否需要代码审查
  };
}
```

**使用示例**:
```typescript
// 优化前（模糊）
{
  task: "修复前端页面错误"
}

// 优化后（精确）
{
  id: "FE-20260328-001",
  title: "修复 ResumeLibrary 页面 TypeScript 错误",
  description: "修复 TDesign 组件类型不匹配问题",
  input: {
    files: ["frontend-web/src/pages/ResumeLibrary/index.tsx"],
    context: "TDesign 5.0 版本 API 变更",
    constraints: ["不改变 UI", "保持兼容性"]
  },
  output: {
    deliverables: ["修复后的文件"],
    format: "TypeScript 代码",
    deadline: "30 分钟"
  },
  execution: {
    timeout: 30,
    parallel: true,
    dependencies: []
  },
  quality: {
    acceptanceCriteria: [
      "TypeScript 编译通过",
      "无类型错误",
      "功能正常"
    ],
    testRequired: false,
    documentationRequired: false,
    codeReviewRequired: true  // 镇抚司审查
  }
}
```

### 1.3 代码审查流程（天策府主导）

**审查流程**:
```
开发团队提交代码（神机营/镇抚司）
    ↓
天策府自动审查（ESLint + TypeScript）
    ↓
天策府人工审查（代码质量 + 安全性）
    ↓
审查通过 → 合并到 develop
    ↓
审查不通过 → 返回修改（附审查意见）
```

**审查清单**:
- [ ] TypeScript 编译通过
- [ ] ESLint 检查通过
- [ ] 单元测试通过
- [ ] 无安全漏洞（JWT_SECRET、数据库密码等）
- [ ] 代码符合规范（命名、注释、结构）
- [ ] 性能无明显问题

**说明**: 天策府负责审查，神机营和镇抚司都是被审查的开发团队。

## 四、子代理并行执行最佳实践

### 4.1 并行度计算

```typescript
// 最优并行度 = min(CPU 核心数，任务数，内存/单任务内存)
const optimalParallelism = Math.min(
  os.cpus().length,
  tasks.length,
  Math.floor(os.totalmem() / (1024 * 1024 * 512)) // 假设每任务 512MB
);
```

### 4.2 任务分组策略

```typescript
// 按依赖关系分组
const groups = {
  independent: tasks.filter(t => !t.dependencies), // 可立即并行
  dependent: tasks.filter(t => t.dependencies),    // 等待依赖
};

// 执行顺序
await executeParallel(groups.independent);
await executeSequential(groups.dependent);
```

### 4.3 资源隔离

```yaml
# 每个子代理的资源限制
resources:
  memory: 512MB    # 防止内存溢出
  timeout: 30min   # 防止无限执行
  cpu: 0.5         # 防止 CPU 独占
```

### 4.4 并行执行模板

```typescript
// 主 Agent 分配任务
const tasks = [
  {
    id: 'FE-001',
    title: '前端页面修复',
    agent: 'frontend-dev',
    parallel: true
  },
  {
    id: 'BE-001',
    title: '后端 API 开发',
    agent: 'backend-dev',
    parallel: true
  },
  {
    id: 'QA-001',
    title: '代码审查',
    agent: 'code-reviewer',
    parallel: false,
    dependencies: ['FE-001', 'BE-001']
  }
];

// 并行执行
const results = await Promise.all(
  tasks.filter(t => t.parallel).map(t => executeTask(t))
);

// 串行执行依赖任务
for (const task of tasks.filter(t => !t.parallel)) {
  await executeTask(task);
}
```
