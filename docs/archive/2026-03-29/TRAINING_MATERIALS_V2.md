## 🎯 第二部分：三源验证机制（15 分钟）

### 核心原则

```
汇报前必须验证三个信息源：
1. 任务看板 (.task-board.md)
2. git 提交记录 (git log)
3. 代码文件检查 (ls/find)

三者一致 → 可汇报
三者不一致 → 先核查再汇报
```

### 核查流程

```bash
# Step 1: 检查任务看板
cat .task-board.md | grep "DEV-001"

# Step 2: 检查 git 提交
git log --grep="DEV-001" --oneline

# Step 3: 检查代码文件
ls -la frontend-web/src/components/SearchBox/

# Step 4: 判断一致性
看板说"待开始" + git 有提交 + 文件存在 = 不一致！
→ 先更新看板，再汇报
```

### 常见不一致场景

| 场景 | 看板 | git | 文件 | 处理 |
|------|------|-----|------|------|
| 已完成但未更新 | 待开始 | 有提交 | 存在 | 更新看板 |
| 进行中但忘提交 | 进行中 | 无提交 | 部分 | 立即提交 |
| 完全同步 | 已完成 | 有提交 | 存在 | ✅ 可汇报 |

## 🎭 第四部分：模拟汇报演练（15 分钟）

### 演练场景 1：任务完成后汇报

**场景**：你刚完成 DEV-006（用户管理模块）

**步骤**：
1. 运行核查脚本
2. 检查三者一致性
3. 更新文档（如需要）
4. 使用模板汇报

**演练**：
```bash
# 1. 核查
./scripts/verify-task-status.sh DEV-006

# 2. 假设输出显示"已完成"

# 3. 汇报（使用模板）
## 📊 DEV-006 用户管理模块 状态

**核查结果**:
- [x] 任务看板：✅ 已完成
- [x] git 提交：✅ 找到提交 `abc123`
- [x] 代码文件：✅ 文件存在（backend/src/modules/user/）
- [x] 文档同步：✅ TODAY_PROGRESS 已更新

**结论**: 功能已完成，提交者：都统，时间：3/29 10:30
```

### 演练场景 2：发现不一致

**场景**：核查发现看板显示"待开始"，但 git 有提交

**步骤**：
1. 立即停止汇报
2. 通知执行者更新看板
3. 等待看板更新
4. 重新核查
5. 然后汇报

**演练**：
```bash
# 1. 核查发现不一致
./scripts/verify-task-status.sh DEV-007
# 输出：⚠️ 任务看板显示"待开始"，但 git 有提交

# 2. 通知执行者
@执行者 DEV-007 看板状态与 git 不一致，请更新

# 3. 等待更新后重新核查
./scripts/verify-task-status.sh DEV-007
# 输出：✅ 三者一致

# 4. 然后汇报
```

### 演练场景 3：每日进度汇报

**场景**：每日站会汇报今日进展

**步骤**：
```bash
# 1. 运行汇报前检查
./scripts/pre-report-check.sh

# 2. 基于输出汇报

- DEV-001 搜索前端 (e147dd9)

正在进行:
- DEV-004 数据看板 (预计 12:00 完成)

核查状态:
✅ 所有汇报已通过三源验证
```
