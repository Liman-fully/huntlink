## 核心原则

### 🎯 五大核心原则

| 原则 | 说明 | 违规后果 |
|------|------|---------|
| **三源验证** | 汇报前必须验证：任务看板 + git 提交 + 代码文件 | 事故报告 + 绩效扣减 |
| **Git 为准** | 一切以 git 提交记录为准，文档必须同步 | 强制更新文档 |
| **自动优先** | 自动化 > 人工操作，能脚本化必须脚本化 | 限期整改 |
| **责任到人** | 每个任务必须有明确的实际执行者 | 责任追溯 |
| **透明可查** | 所有状态公开透明，随时可查 | 通报批评 |

### 🔴 零容忍行为

1. ❌ 未三源验证就汇报
2. ❌ 任务完成后不更新文档
3. ❌ 提交信息不包含任务 ID
4. ❌ 标注虚假执行者
5. ❌ 隐瞒事故不报

### 任务状态

| 状态 | 代码 | 说明 | 转换条件 |
|------|------|------|---------|

```

  领取任务    开始执行    git 提交 + 文档更新
```

## 状态同步

### 三源验证机制

```
┌─────────────────────────────────────────────────────────┐
│                    任务状态汇报                          │
├─────────────────────────────────────────────────────────┤
│  1️⃣ 任务看板 (.task-board.md)                           │
│  2️⃣ git 提交记录 (git log)                               │
│  3️⃣ 代码文件检查 (ls/find)                               │
└─────────────────────────────────────────────────────────┘
                        ↓
            【三者一致】→ 可汇报
            【三者不一致】→ 先核查再汇报
```

### 强制核查清单（CROSS-VERIFY）

**C - Check git log（检查提交记录）**
```bash
git log --oneline -10
git log --all --oneline --grep="任务 ID" | head -5
```

**R - Read task board（读取任务看板）**
```bash
cat .task-board.md | grep "任务 ID"
cat docs/TODAY_PROGRESS.md | grep "任务 ID"
```

**O - Observe code files（检查代码文件）**
```bash
ls -la frontend-web/src/pages/Dashboard/ 2>/dev/null || echo "文件不存在"
ls -la backend/src/modules/[模块名]/ 2>/dev/null || echo "模块不存在"
```

**S - Synchronize documents（同步文档）**
```bash
# 如果不一致，立即更新文档
# 以 git 提交为准
```

**S - Verify with team（与团队确认）**
```bash
# 如有疑问，@相关成员确认
# 不要假设，不要猜测
```

### 文档同步要求

| 文档 | 更新时机 | 负责人 | 检查方式 |
|------|---------|--------|---------|
| .task-board.md | 任务完成后立即 | 执行者 | CI/CD 检查 |
| docs/TODAY_PROGRESS.md | 任务完成后立即 | 执行者 | CI/CD 检查 |
| docs/DEV_TASKS_*.md | 任务启动时 | 执行者 | 人工检查 |
| README.md | 每周 | 协调者 | 周会检查 |

## 子代理管理

### 子代理创建规范

```yaml
# 子代理任务分配模板
任务 ID: DEV-xxx
任务名称：[简短描述]
实际执行者：[左护法/右护法/都统]
子代理标签：[执行者]-[任务 ID]-[任务名]
预计时间：[分钟]
交付物：
  - [文件 1]
  - [文件 2]
验收标准:
  - [标准 1]
  - [标准 2]
```

### 子代理状态追踪

| 状态 | 说明 | 处理方式 |
|------|------|---------|
| 已启动 | 子代理已创建 | 记录到追踪表 |
| 执行中 | 子代理正在执行 | 定期询问进度 |
| 已完成 | 子代理完成报告 | 立即更新任务看板 |
| 失败 | 子代理执行失败 | 分析原因，重新分配 |

### 子代理完成处理流程

```
1. 子代理完成报告 → 主 session
2. 主 session 确认交付物
3. 执行 git commit + push
4. 更新任务看板（标记为已完成）
5. 更新 TODAY_PROGRESS.md
6. 通知相关成员
7. 关闭子代理任务
```

### 责任绑定机制

| 角色 | 责任 |
|------|------|
| **创建者** | 创建子代理，分配任务，验收结果 |
| **执行者** | 实际执行子代理任务（可能是 AI） |
| **标注者** | 任务看板上标注的负责人 |

**原则**: 创建者 = 标注者 = 责任承担者

## Git 规范

### 提交信息格式

```
<类型>(<任务 ID>): <简短描述>

<详细说明（可选）>

交付物:
- [文件 1]
- [文件 2]

关联任务：[任务 ID]
```

### 提交类型

| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | feat(DEV-005): 实现人才推荐算法 |
| fix | Bug 修复 | fix(BUG-001): 修复部署验证脚本 |
| docs | 文档更新 | docs: 更新任务看板 |
| refactor | 重构 | refactor: 优化缓存服务 |
| test | 测试 | test: 添加缓存测试脚本 |
| chore | 构建/工具 | chore: 更新依赖 |

### 推送要求

| 操作 | 时机 | 要求 |
|------|------|------|
| commit | 每完成一个小功能 | 必须包含任务 ID |
| push | commit 后立即 | 不得本地堆积 |
| pull | 开始工作前 | 获取最新代码 |
| merge | 功能完成后 | PR 审查后合并 |

### 分支策略

```
master (生产)
  ↑
develop (开发)
  ↑
feature/DEV-xxx (功能分支)
```

## 汇报机制

### 汇报频率

| 汇报类型 | 频率 | 时间 | 汇报人 |
|---------|------|------|--------|
| 进度汇报 | 每 2 小时 | 整点 | 各主官 |
| 日报 | 每日 | 23:00 | 左护法 |
| 周报 | 每周 | 周日 20:00 | 左护法 |
| 事故报告 | 立即 | 事故发生后 30min 内 | 责任人 |

### 汇报模板

```markdown
## 📊 [功能名] 状态

**核查结果**:
- [x] 任务看板：✅ 已完成
- [x] git 提交：✅ 找到提交 `xxx`
- [x] 代码文件：✅ 文件存在
- [x] 文档同步：✅ 已更新

**结论**: 功能已完成，提交者：xxx，时间：xxx
```

### 禁止的汇报方式

❌ **错误示例 1**: 仅看任务看板
```markdown
## 📊 DEV-001 状态
**状态**: ⏳ 待开始（任务看板显示）
```

❌ **错误示例 2**: 未三源验证
```markdown
## 📊 DEV-005 状态
**状态**: 没有开发（右护法说）
```

✅ **正确示例**: 三源验证
```markdown
## 📊 DEV-005 状态
**核查结果**:
- [x] 任务看板：✅ 已完成
- [x] git 提交：✅ 找到提交 `6f32be6`
- [x] 代码文件：✅ 5 个文件，940 行
- [x] 文档同步：✅ 已更新

**结论**: 功能已完成，提交者：左护法，时间：00:20
```

## 事故处理

### 事故分级

| 级别 | 说明 | 响应时间 | 处理人 |
|------|------|---------|--------|
| P0 | 系统崩溃、数据丢失 | 5 分钟 | 全体 |
| P1 | 状态追踪失误、错误汇报 | 30 分钟 | 责任人 + 主官 |
| P2 | 文档不同步、提交不规范 | 2 小时 | 责任人 |
| P3 | 轻微违规、流程瑕疵 | 24 小时 | 责任人 |

### 事故报告模板

```markdown
# 🔴 事故报告 #XXX - [事故名称]

**报告时间**: YYYY-MM-DD HH:MM
**报告人**: [姓名]
**事故级别**: P0/P1/P2/P3
**影响范围**: [描述]

## 📋 事故描述
[详细描述]

## 🔍 根本原因分析
[分析]

## 📊 影响评估
[评估]

## ✅ 纠正措施
[措施]

## 📝 改进计划
[计划]

## 🙏 责任承担
[责任]
```

### 事故处理流程

```
1. 发现事故 → 立即报告
2. 评估级别 → 确定处理人
3. 纠正措施 → 立即执行
4. 根因分析 → 防止再发
5. 经验总结 → 记录到文档
6. 团队通报 → 全员学习
```
